最近我在本地博客项目里做了一件很有意思的事：不是直接让 AI 帮我写代码，而是先让它帮我沉淀一个可以复用的工作流，也就是 Codex Skill。

这次的目标很具体：创建一个“写博客并一键部署”的技能。它要知道当前项目是 Hexo 博客，文章应该放在哪里，写作前要怎样确认大纲，生成后要怎样验证，什么时候可以部署，什么时候必须停下来等我确认。

这篇文章整理的就是这次对话里的完整学习过程。

一、Skill 不是提示词堆砌

一开始很容易把 Skill 理解成一段更长的提示词，比如：

你是一个博客助手，请帮我写高质量文章，并部署到线上。

这类描述看起来正确，但真正执行时不够可靠。因为它没有告诉 AI：

  • 当前项目是什么框架
  • 文章应该写到哪个目录
  • frontmatter 应该是什么格式
  • 图片资源应该怎么放
  • 部署命令是什么
  • 部署前是否需要确认

好的 Skill 更像一份轻量的项目操作手册。它不需要解释通用常识，而是要补上 AI 默认不知道、又容易做错的项目细节。

三、把需求先变成计划

在真正写 Skill 之前，先把需求拆成了几个关键决策：

  • Skill 放在哪里
  • 是否要全局生效
  • 部署是自动执行，还是确认后执行
  • 写作流程是直接成文，还是先大纲后成文

最终选择是：

  • 放在当前仓库的 skills/ 目录
  • 专门服务这个 Hexo 博客
  • 写作时先确认标题、大纲和标签
  • 部署前必须先构建验证，再等待用户明确确认

这个过程很重要。因为 Skill 一旦写好，后续 AI 会按它行动。如果这些规则没有提前说清楚，最容易出问题的就是“误发布”和“写到错误位置”。

正文部分则写入了几类规则。

1. 仓库事实

Skill 里明确记录：

  • 当前项目是 Hexo 6
  • 文章放在 source/_posts/
  • 使用 title、date、tags 形式的 frontmatter
  • 已开启 post_asset_folder
  • 部署命令等价于 hexo clean && hexo g && hexo d
  • 本地预览习惯使用 8080 端口

这些都是从项目里观察出来的，不是猜出来的。

2. 写作流程

如果用户只给一个主题，Skill 要求先生成：

  • 标题
  • 大纲
  • 标签

确认方向后，再创建 Markdown 文件。

如果用户给的是笔记或草稿，Skill 要求保留用户原意，整理结构，补全 frontmatter、标题、标签和结尾。

4. 验证与部署

部署是最需要谨慎的地方。

Skill 里明确写了：

  1. 修改文章或站点输出后，先运行 npm run build
  2. 构建成功后，才能考虑部署
  3. 部署前必须让用户明确确认
  4. 确认后才执行 hexo clean && hexo g && hexo d

也就是说，“一键部署”不是无脑自动部署，而是把构建、检查、确认、发布这条链路规范化。

五、为什么要加确认门槛

博客部署看起来不是高危操作，但它仍然会影响线上内容。

如果没有确认门槛，AI 可能在这些情况下误发布：

  • 文章只是草稿，还没校对
  • 标签或标题还没确认
  • 构建有警告但没有被注意
  • 用户只是想预览，并不想上线
  • 当前仓库还有未处理的其他改动

所以这次 Skill 的核心原则是：

写作可以主动，部署必须克制。

这不是降低效率，而是把风险最高的一步从自动化里拎出来，让用户最后拍板。

六、验证 Skill 是否写对

创建完成后，还做了几项检查：

  • SKILL.md 是否有合法的 name 和 description
  • 是否覆盖“写博客”“整理笔记”“预览博客”“部署博客”等触发场景
  • 是否明确写了构建验证
  • 是否明确写了用户确认后才能部署

这些检查不复杂，但可以避免 Skill 变成一篇“看起来很完整，实际不好用”的文档。

七、我对生成 Skill 的理解

这次实践之后，我对 Skill 的理解更清晰了：

Skill 不是让 AI 更“聪明”，而是让 AI 更“熟悉这里”。

它应该沉淀的是项目里的真实约定：

  • 目录结构
  • 命令习惯
  • 安全边界
  • 输出格式
  • 容易犯错的地方
  • 用户偏好的工作方式

尤其是本地开发和部署类任务，Skill 里一定要写清楚哪些事情可以自动做，哪些事情必须确认。

八、一个可复用的 Skill 生成流程

以后再创建类似 Skill，可以按这个流程走：

  1. 先观察项目结构，不要凭空写
  2. 找到真实入口，比如 package.json、配置文件、脚本、已有文档
  3. 把用户目标拆成工作流
  4. 明确输入、输出和文件位置
  5. 写清楚验证命令
  6. 给危险动作加确认门槛
  7. 最后用几个真实触发句测试 Skill 是否会被正确调用

如果一个 Skill 能做到这些，它就不只是“提示词”，而是一份可以被反复使用的项目经验。

总结

这次从一句“我需要创建一个写博客并一键部署的技能”开始，最终沉淀出了一个专门服务当前 Hexo 博客的 hexo-blog-publisher Skill。

整个过程里最有价值的不是生成了一个文件，而是把模糊需求变成了明确流程：

  • 写作先确认方向
  • 文章放到正确目录
  • 资源遵循 Hexo 约定
  • 发布前先构建
  • 部署前必须确认

这就是我觉得 Skill 最有用的地方：把一次对话里的经验，变成下一次可以直接复用的能力。
